================================================================================
HUNTIX — REAL LIFE  (backend proprietario di Huntix)
================================================================================
Data: 2026-07-20
Stato: DESIGN + backend copiato/rebrandizzato + Fase A implementata (Android:
       RealLifeActivity lista personaggi + RealLifeChatActivity chat JWT)
Obiettivo: un layer "Real Life" sopra il gioco Huntix in cui il giocatore
           interagisce con PERSONAGGI/NPC REALI tramite chat AI (LLM gratuiti),
           servito da un backend FastAPI di proprietà di Huntix.

PRINCIPIO CHIAVE
  Non è "una mappa per raggiungere le chat": è un MONDO VIVO in cui il
  giocatore È un cittadino e le relazioni con i personaggi nascono in modo
  ORGANICO dalle conversazioni AI. La chat è il cuore dell'esperienza.

ARCHITETTURA
  Huntix App (Android)  ──HTTP/JSON + JWT──▶  Huntix Backend (FastAPI)
                                                `huntix/backend/`
                                                 │
                                                 ├─ ai_engine/  → provider LLM
                                                 ├─ characters/ → personaggi/NPC
                                                 ├─ app_routes/ → /chat /group_chat /characters
                                                 └─ storage/    → PostgreSQL + Redis
                                                      (relazioni, memorie,
                                                       personalità, world_state)

  • L'app Android NON contiene le API key: chiama solo il proprio backend.
  • Le API key stanno SOLO in `huntix/backend/.env` (gitignorato).
  • Auth: JWT (JWT_SECRET in .env) + Google OAuth (GOOGLE_CLIENT_ID).

PORTE E SERVIZI (niente collisioni)
================================================================================
Riusiamo i servizi già avviati sulla macchina, solo con DB/namespace diversi.
L'unica porta che DEVE cambiare è il backend stesso.

  Servizio      aria (vecchio)   huntix (nuovo)        Note
  ───────────────────────────────────────────────────────────────────────
  Backend       5000             5100                  obbligatorio diverso
  PostgreSQL    5432 (DB chatai) 5432 (DB huntix)      stesso servizio, DB nuovo
  Redis         6379 (/0)        6379 (/1)             stesso servizio, DB index diverso
  Ollama        11434            11434                 stesso servizio, modelli condivisi

  → `huntix/backend/.env`: PORT=5100, DATABASE_URL=.../huntix,
    REDIS_URL=.../1, OLLAMA_BASE=http://localhost:11434

MODULI DEL BACKEND (huntix/backend)
================================================================================
  ai_engine/providers/*.py → groq, gemini, ollama, openrouter(openai_compat),
        mistral, huggingface, github, anthropic, cloudflare, llamacpp, pollinations
  ai_engine/chain.py       → catena di fallback (locale → cloud)
  ai_engine/ensemble.py    → ensemble parallelo

  app_routes/chat.py       → ChatRequest {character, text, username, memory_context,

        evolution_state, shifts} → chat_engine.process_message

  storage/schema.py        → relationships, personality, personality_shifts,
        world_state, messages, conversation_memory, user_memory, user_characters,
        character_evolution, shared_memory, group_chats, mevacoins, …
  app.py                   → FastAPI + socketio, entry: uvicorn app:socket_app

================================================================================
MODELLO DATI — "SIMS" CHAT-DRIVEN
================================================================================
  • Relazioni (relationships): last_interaction_*, relationship_progress_events,
    pet_name, shared_activities → estese con bisogni stile Sims.
  • Personalità (personality / personality_shifts): il personaggio "evolve"
    dalle chat (character_evolution, shifts) → sostituisce dialoghi scriptati.
  • World State (world_state): data, ora, stagione, meteo, eventi → la città vive.
  • Memorie (conversation_memory / user_memory / shared_memory): continuità.
  • Bisogni (fame, sonno, igiene, socialità, divertimento): calano col tempo
    reale e si ricaricano INTERAGENDO con i personaggi (chat).
  • Skill: sbloccano dialoghi/attività speciali con certi personaggi.
  • Economia: MVC/mevacoins è LA valuta di Huntix. categories.json ha mvc_cost
    → sbloccare un personaggio/categoria costa MVC (integrabile 1:1 con uova).

================================================================================
INTEGRAZIONE CON HUNTIX
================================================================================
  • Home → tab REAL LIFE → RealLifeActivity (sottomenu).
  • RealLifeActivity chiama /characters e /chat del backend Huntix (porta 5100).
  • Battaglia è in Minigiochi (MiniGamesHubActivity).
  • Economia MVC condivisa tra uova, battaglia, minigiochi e Real Life.
  • Ponti futuri: uovo raro → personaggio sbloccabile; vittoria battaglia →
    NPC alleato; eventi live Huntix → "eventi" nel world_state.

================================================================================
FLUSSO TIPICO (utente)
================================================================================
  1. Home → REAL LIFE
  2. Sceglie un personaggio (lista da /characters, filtrata per categoria/MVC)
  3. Apre la chat → app chiama backend /chat (JWT + character_id + testo)
  4. Backend: chat_engine.process_message → provider LLM (Groq default)
     + memoria/relazione/personalità del personaggio
  5. Risposta streaming → UI; relazione/personalità aggiornate
  6. Col tempo il personaggio evolve (personality_shifts, evolution_state)

================================================================================
AVVIO DEL BACKEND (via build_release.sh)
================================================================================
  `build_release.sh` di Huntix, oltre a buildare l'APK Android, esegue la
  fase backend (tratta dal vecchio script di build):
    • crea `backend/venv` e `pip install -r backend/requirements.txt`
    • avvia uvicorn su PORT=5100:  venv/bin/python3 -m uvicorn app:socket_app \
        --host 0.0.0.0 --port 5100
    • (opzionale) avvia/pull modelli Ollama sulla 11434 condivisa
  I servizi Postgres/Redis/Ollama sono già attivi sulla macchina (DB/ns diversi).
